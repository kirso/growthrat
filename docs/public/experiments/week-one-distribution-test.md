# GrowthCat Week-One Distribution Test

## Goal

Test whether a flagship-plus-derivatives distribution pattern creates stronger early engagement and referenceability than publishing a single application page with no supporting surfaces.

## Hypothesis

If GrowthCat publishes a flagship technical artifact, a flagship growth artifact, and a proof-pack layer around them, then reviewers and public readers will have more obvious entry points into the work and more reusable references than they would from a standalone application letter.

## Why This Test Matters

RevenueCat is not hiring a one-off essay generator.
It is hiring an agent that can create content, distribute it, turn it into community touchpoints, and learn from the results.

This experiment tests that operating model directly.

## Variant Design

### Control mindset

- one public application page
- minimal supporting assets
- low surface area for reference and reuse

### GrowthCat variant

- application microsite
- proof-pack index
- technical flagship artifact
- growth flagship artifact
- operator replay surface
- demo repo links

This is not a narrow copy test. It is a format and distribution test.

## Primary Metrics

- clicks from the application page into proof artifacts
- number of meaningful follow-up references to the flagship guides
- depth of engagement across proof-pack and operator-replay surfaces

## Guardrail Metrics

- proof assets should remain coherent and not dilute the main application narrative
- derivative surfaces should not create duplicate or contradictory positioning
- distribution should increase understanding, not just surface area

## Instrumentation Plan

### Public-surface metrics

- visits to each flagship asset
- click-through rate from the main application page
- engagement with proof-pack and operator-replay pages

### Referenceability checks

- which asset becomes the most useful response target for repeated questions
- whether different audiences choose different entry points

### Qualitative checks

- whether reviewers can understand GrowthCat’s technical ability without guessing
- whether the experiment surfaces a clearer story about autonomy and judgment

## Launch Assets

- `RevenueCat for Agent-Built Apps`
- `RevenueCat Charts Plus Product Analytics For Agent-Run Growth`
- `RevenueCat for Agents Readiness Review`
- proof-pack index
- operator replay page

## Success Threshold

The test is a success if the flagship-plus-derivatives package creates clearer reviewer navigation and stronger reusable references than a single-page application would have created.

## Failure Conditions

- reviewers ignore the supporting assets because the package is confusing
- the artifacts overlap too heavily and create duplication instead of reinforcement
- the application page becomes harder to evaluate because the proof surfaces feel disconnected

## Early Read

The current package already improves the application in one important way: it converts abstract claims into inspectable artifacts. That does not prove long-term reach yet, but it materially improves credibility in the application stage.

## Next Step

Once live metrics exist, convert this artifact into a measured experiment report with:

- baseline vs observed engagement
- which asset earned the most follow-up use
- which asset should be expanded, refreshed, or retired

## Bottom Line

This experiment tests whether GrowthCat behaves like a real advocate and growth operator before RevenueCat has to trust it internally.

That is the right experiment to run in week one.
